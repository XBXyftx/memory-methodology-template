# Agent 工作流优化内容摘要

> 本文是一次 Agent 工作流优化的脱敏总结，可作为其他项目迁移长期记忆体系和 sub agent 机制时的参考。

## 1. 优化背景

原工作流存在以下典型问题：

- 每次任务都要求重复读取大量长期记忆文档，启动成本较高。
- 部分 sub agent 被设计为固定步骤，简单任务也会触发不必要的上下文分析。
- 主 Agent、sub agent 和文档写作规则之间存在重复内容。
- 不同 AI 客户端的 Agent 配置容易逐渐产生语义差异。
- Agent 输出缺少统一的证据分类和停止条件，容易把推断写成事实。

## 2. 优化目标

本次优化围绕以下目标展开：

1. 保留必要的启动约束，但减少重复读取。
2. 根据任务复杂度选择最小的 sub agent 集合。
3. 统一主 Agent 与 sub agent 的生命周期、输入和输出格式。
4. 区分当前事实、历史结论、推断、用户反馈和待验证事项。
5. 让多个 AI 客户端共享同一套 Agent 提示词来源。
6. 保持长期记忆文档短小、可导航、可持续维护。

## 3. 文档职责分层

将工作流拆分为以下职责层，避免规则互相复制：

| 文档层 | 主要职责 |
|---|---|
| 工作区启动指令 | 项目边界、授权限制、基础读取协议 |
| 全局长期记忆索引 | 当前状态、全局导航和重要结论 |
| 项目规则 | 代码修改、验证、授权和安全边界 |
| Agent 工作协议 | Agent 生命周期、证据标签、输出格式和停止条件 |
| Sub agent 路由矩阵 | 任务类型与最小 Agent 组合的映射 |
| 文档编写规则 | 文件命名、分层、拆分、索引和归档规范 |
| Agent 专项提示词 | 单个 Agent 的职责和专项检查项 |

其中，Agent 工作协议只定义共性行为，路由矩阵只定义选择逻辑，专项提示词只保留角色差异。

## 4. 统一 Agent 生命周期

### 4.1 接收与分类

主 Agent 在开始工作时先明确：

```text
任务类型：BUG / REQ / Code Review / 文档维护 / 只读分析 / 其他
任务目标：用户希望得到什么结果
任务范围：仓库、模块、文件或文档入口
现有证据：现象、日志、截图、测试反馈或源码线索
写入范围：只读或允许修改的目录
结束条件：什么事实或验证结果可以结束任务
```

当目标文件、函数、修改方式和验收条件都明确，且没有历史不确定性、工作区风险或跨模块影响时，主 Agent 直接处理，不启动 sub agent。

### 4.2 渐进式读取

- 所有任务先读取工作区启动指令、全局索引和项目规则。
- 需求、Bug、API、Code Review 或架构任务只进入对应的目录入口。
- 文档编写规则只在新增、移动、拆分、归档或实质更新文档时读取。
- 领域 skill 只在需要框架、语言、工具链或错误分析指导时加载。

### 4.3 按需编排

- 先选择最小的 Agent 集合。
- 可以并行的 Agent 在同一批次并行执行。
- 结果已经回答问题时立即停止。
- 最多追加一批 Agent，除非用户明确要求深度审查。
- 不因“涉及代码”“涉及框架文件”或“准备交付”自动启动全量 Agent。

### 4.4 证据合并

统一使用以下标签：

| 标签 | 含义 |
|---|---|
| `代码事实` | 当前源码、配置或 diff 直接证明的内容 |
| `历史结论` | 长期记忆或旧方案中的记录，需要核对当前源码 |
| `用户/测试反馈` | 用户、测试人员或人工环境提供的结果 |
| `推断` | 基于证据形成的解释，尚未完全证实 |
| `待确认` | 缺少源码、日志、测试数据或用户确认 |
| `已验证` | 有明确验证结果支持的结论 |

不得把 Agent 建议写成已加载 skill，不得把静态分析写成编译或设备验证结果，也不得把提交记录当作测试证据。

### 4.5 执行与收尾

- 修改前先形成最小改动计划。
- 修改后检查 diff、临时诊断、索引链接和受保护差异。
- 记录实际执行的静态检查，并单独列出人工或设备验证项。
- 文档变更时同步条目入口和父级索引。
- 未经授权不执行提交、推送、恢复、删除或外部系统写操作。

## 5. 按需选择 sub agent

推荐使用“问题信号 → 最小角色 → 追加条件”的方式，而不是按任务名称固定启动。

| 问题信号 | 最小角色 | 追加条件 |
|---|---|---|
| 历史入口不清 | 记忆路由 Agent | 找到相关入口后停止 |
| 目标文件有本地差异 | 工作区保护 Agent | 只检查指定文件和 diff |
| 新行为或复用关系不清 | 现有链路 Agent | 发现跨模块后追加架构 Agent |
| Bug 调用链不清 | Bug 路径 Agent | 有日志时追加日志分析 Agent |
| 已有日志、崩溃报告或多步复现 | 日志分析 Agent | 需要源码对应时追加路径 Agent |
| 跨模块、全局状态或公共服务变化 | 架构 Agent + 链路 Agent | 根据任务类型追加 Bug 或验证 Agent |
| 框架代码涉及状态、生命周期、手势或 API 风险 | 合规审查 Agent | 纯样式和资源调整跳过 |
| 复杂改动需要人工回归 | 验证规划 Agent | 影响面明确后再启动 |
| 多个文档需要同步交接 | 交接摘要 Agent | 单行索引修改跳过 |

典型任务结论：

- 纯颜色、尺寸、文案、资源或单文件参数调整：通常不启动 sub agent。
- 有日志的运行时 Bug：优先日志分析，只有调用链不清时再追加路径追踪。
- 跨模块需求：优先现有链路和架构分析，确认影响面后再决定是否生成验证矩阵。
- 已明确文件和改法的简单需求：由主 Agent 直接处理。

## 6. 统一输出模板

每个 sub agent 使用以下输出顺序，只保留与职责相关的内容：

```text
适用性：适用 / 不适用；命中的触发条件
任务边界：本次检查了什么，没有检查什么
代码事实：路径、函数、字段、调用关系和证据位置
历史结论：有效、废弃或与当前事实冲突的记录
风险与待确认：证据缺口、潜在影响和需要补充的信息
建议下一步：主 Agent 需要执行的最小动作
停止原因：证据足够 / 不适用 / 缺少外部证据 / 达到范围边界
```

## 7. 多客户端同步原则

- 为每个 Agent 保留一个主提示词源。
- 不直接手工维护多个客户端的完整提示词副本。
- 迁移前先使用隔离源目录执行 dry-run。
- 迁移后检查生成文件的名称、描述、开发者指令和工具语义。
- 验证生成目标时，不得覆盖工作区启动指令或无关配置。
- 客户端之间无法完全等价的权限、模型、skill 预加载和生命周期能力，应记录为适配差异。

## 8. 可复用收益

这套方式适合以下场景：

- 长期记忆文档数量较多，需要降低新会话启动成本。
- 同一项目同时使用多个 AI 客户端。
- 项目包含代码、Bug、需求、日志和文档归档等不同类型任务。
- 希望使用 sub agent 提升定位速度，但不希望简单任务消耗额外资源。
- 需要让 AI 输出始终区分证据、推断和待验证项。

## 9. 迁移检查清单

- [ ] 工作区启动指令没有要求默认启动全量 Agent。
- [ ] 全局索引包含 Agent 工作协议和路由入口。
- [ ] 任务路由包含“不启动条件”和最小组合。
- [ ] Agent 提示词包含适用性判断和停止条件。
- [ ] Agent 输出能够区分事实、推断、反馈和待确认项。
- [ ] 文档写作规则没有重复承载 Agent 调度逻辑。
- [ ] 多客户端使用同一提示词源。
- [ ] 生成目标通过格式和字段校验。
- [ ] 新增文档已完成脱敏、链接和长度检查。
